加密

对称加密

  • 应用场景

    • 文件加密、数据加密。

  • 密钥分发困难、不安全但是速度快、效率高

  • 解决方法

    • 解密端与加密端，要确保这两个密钥密钥一样，不能直接发送密钥
    • 而是各自发送一部分字符给对方，两者结合，再用固定的算法生成相同的密钥
    • 解密端与加密端要建立通路 —— 套接字通信（用来分发密钥）

  • 相同的密钥产生之后是用来加密的，也就是说生成之后需要存储

    • 若是存在数据库，则打开、关闭数据库，效率太低（多次访问的话）
    • 应该存在内存，然后应用程序在内存之间读取，实质是进程间通信！
      • 管道、信号、mmap、socket
      • mmap 最简单，直接用共享内存、提高访问效率

  • 既然是对称加密，则两端要定期生成随机数、算法也要换！

    • 但是老的密钥要保存，这里可以用到数据库（持久化存储）、必须得做

  • 应用场景

    • 进行身份鉴别、密钥保护。

  • 为什么要进行这一步？

  • 客户端要发送随机数给服务端、服务端也要发送给随机数给客户端；两者结合形成密钥

编码

  • TLV 格式
    • 基本上是通过值-结果参数（指针的指针）来获得编码后的结果
    • 而函数的返回值（整形）是用来判断编码是否成功，例如分配内存报错等等
    • 若报错，则用日志库的函数进行输出记录！
    • 这里面有个技巧，若每次判断返回值是否成功都写在主函数里，会变得啰嗦，可以再写一个包裹函数（包括了判断），直接调用包裹函数即可

解码

  • 就是将一整个 TLV 嵌套的数据类型依次取出对应的数据
  • 因为原数据是一个结构体，里面有几种类型的数据，在此之间要先给结构体分配空间
    • 若结构体内有指针（没有对应指向的空间）那么默认会分配一个字节的大小用来储存地址，这时另需分配额外的空间来存储（解码后的）内容

内存释放

  • 这里有个技巧

    • 一般来说，大多数函数里都会进行分配内存、释放内存的操作（手动调用），这些在函数外部是看不出来的
    • 主要是从函数的参数（传出参数、指针的指针）进行判断
      • 若传出失败，只需要判断其余参数是否需要释放内存即可

  • 不用了就释放

    • 除了结构体本身占据一定空间之外，包含的指针也会指向一片空间，需要释放
    • 有点类似于释放链表，但是节点也有指针需要释放

  • 同时需要注意临时变量以及要输出的变量

    • 若输出成功，则临时变量开辟的空间也需要释放

  • 有一个问题

    • void freeXXX (T *t)、void freeXXX (T **t) 哪个好
    • 视频上是说第二个好，因为可以赋值？？？没理解。。。
    • 如果是内置类型的变量，可以直接强制转换为指针，比如 int 类型，传过去之后再强转回来
    • 如果是结构体，那么必须得取地址再转为 void * 或者 void ** 类型

  • 还有，结构体参数一般都是指针传递

共享内存

  • 特征
    • IPC（进程间通信）
    • 管道 pipe：简单，只能在有血缘关系的进程间通信。
    • 命名管道 fifo：在无血缘关系的进程间通信，内部数据只能读一次。
    • mmap：在无血缘关系的进程间通信。内部数据支持反复读取。
    • 信号：开销小，携带数据简单。
    • 本地 socket： 稳定性最好，实现复杂。
    • shared memory ： 类似于 mmap。 效率最高IPC。
  • mmu
    • 物理内存、虚拟内存的映射关系
    • 虚拟地址不存在（哪怕虚拟地址一样，物理地址也不一样（用户空间：0-3G））
    • 内核空间映射：映射的物理地址是一样的，因为只有一个操作系统，也就只有一个内核，通过 PCB（描述符）来区分不同的进程

  • 常用 API
    • int shmget(key_t key, size_t size, int shmflg);
    • void *shmat(int shmid, const void *shmaddr, int shmflg);
    • int shmdt(const void *shmaddr);
    • int shmctl(int shmid, int cmd, struct shmid_ds *buf);
  • 共享内存需要脚本去删（杀掉）
    • 不是，还好，不需要像视频那样 ipcrm -m shm_id 即可
    • ipcs 查看
  • 多个进程使用同一个共享内存的时候
    • 其中一个进程关闭 shmctl() 内存的时候，只是 ‘mark’ 了一下，并没有真正的关闭内存
    • 其次，共享内存的 key 会变为 0x0000 变成私有的，新的进程无法再次使用这片内存
    • 引用计数会减一，为 0 会真正删除这个共享内存。

  • Linux 内核管理共享内存的方法

    • key 值 + 引用计数 技术。

  • 注

    • socket 池是用用来拓展知识面的，实际用不到

    • 编译的时候会出现找不到动态库的情况，这里顺便复习下动态库的几种访问路径或者直接在编译的时候加入 -L xx/xx/ 参数
  • 之后为了方便应该把用到的动态库文件统一放在 ~/lib/ 下，再修改下配置文件即可
  • 关于 MySQL 游标
    • 使用 NOT FOUND 来结束游标当其他查询没有返回结果都会触发造成提前结束
    • 可以先将游标的总记录数查询出来，然后用记录数来确定循环次数就可以了。
    • 对于 MySQL 应该没有那么麻烦，直接 mysql_fetch_row() 就可以一行一行读取了，所有的结果集都在 result 里就很方便
    • 所以这些数据库有关的代码都大大简化了，连接池还是得继续用，事务开启以及关闭要继续使用，显得逻辑清晰
